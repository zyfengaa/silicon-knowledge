# 02 — 流水线冒险总览

> 流水线冒险（Hazard）是指由于后续指令无法在下一个时钟周期正确执行而导致流水线无法按预期推进的情况。冒险是流水线设计中需要解决的核心问题。

---

## 1. 冒险的定义

理想的流水线每周期推进一条指令。但实际中，指令之间的依赖关系或硬件资源的限制可能导致下一条指令无法在下一个时钟周期开始执行。这种"阻碍"就是冒险。

冒险会导致**流水线停顿**（pipeline stall，也称为 bubble）——在流水线中插入一个或多个空操作（NOP），等待依赖条件满足后继续执行。

---

## 2. 三种冒险类型

流水线冒险分为三类：

| 类型 | 根本原因 | 解决思路 |
|------|---------|---------|
| **结构冒险**（Structural） | 硬件资源冲突 | 增加硬件资源 / 分离通路 |
| **数据冒险**（Data） | 指令间的数据依赖 | 转发（forwarding）+ 必要时停顿 |
| **控制冒险**（Control） | 分支等控制流指令 | 分支预测 + 延迟槽 |

---

## 3. 结构冒险

### 3.1 产生原因

两条指令在同一时钟周期需要访问同一个硬件资源。

在我们的 5 级流水线中，IF 阶段需要访问指令存储器，MEM 阶段需要访问数据存储器。如果二者共用同一个存储器（冯·诺依曼结构），就会产生结构冒险。

### 3.2 示例：存储器冲突

```
周期 3:
  lw 指令在 MEM 阶段 → 访问数据存储器
  add 指令在 IF 阶段 → 访问指令存储器（下一条指令）

如果数据存储器与指令存储器是同一个物理存储器，
则产生结构冒险，两条指令无法同时访问。
```

### 3.3 解决方法

1. **分离指令和数据存储器**: 哈佛架构（Harvard architecture），即使用独立的指令缓存（I-cache）和数据缓存（D-cache）
2. **增加资源**: 如多端口寄存器堆
3. **流水线停顿**: 检测到冲突时插入气泡（性能损失）

现代处理器普遍采用分离的 I-cache 和 D-cache 来避免结构冒险。

> 结构冒险在 RISC-V 5 级流水线中**相对少见**，因为 IF 和 MEM 访问不同的存储器。但如果使用单端口寄存器堆（要求同一个周期内读写不同寄存器），WB 写回和 ID 读寄存器可能发生冲突。解决方法是在 WB 阶段前半周期写、ID 阶段后半周期读。

---

## 4. 数据冒险

### 4.1 产生原因

一条指令依赖于前面指令的计算结果，且该结果尚未写回寄存器堆。

数据冒险是最常见、最重要的冒险类型，将在下一节详细讨论。

### 4.2 示例：RAW 冒险（Read After Write）

```
add x1, x2, x3    # 写入 x1
sub x4, x1, x5    # 读取 x1 ← 需要 add 的结果
```

在流水线中：

```
周期:   1      2      3      4      5
add:    IF    ID    EX    MEM    WB    ← x1 在 WB 阶段才写入
sub:          IF    ID    EX    MEM   WB
                      ↑ sub 在 EX 阶段需要 x1
                      x1 尚未写回 → 数据冒险
```

### 4.3 数据冒险的三种类型

| 类型 | 含义 | RISC-V 中出现 | 说明 |
|------|------|--------------|------|
| **RAW** | Read After Write | **常见** | 真正需要解决的依赖 |
| **WAR** | Write After Read | 不会出现（？） | 在按序流水线中，读在 ID 阶段，写在 WB 阶段，不可能先写后读 |
| **WAW** | Write After Write | 不会出现（？） | 在按序流水线中，WB 阶段按指令顺序写回，不可能乱序写 |

> 注：WAR 和 WAW 在**乱序执行**的超标量处理器中会出现。在简单的按序 5 级流水线中，只需关注 RAW 冒险。

### 4.4 解决方法

- **转发（Forwarding / Bypassing）**: 将 EX 或 MEM 阶段的计算结果直接送到 EX 阶段的 ALU 输入端，跳过寄存器写回→读取的延迟
- **流水线停顿**: 当转发无法解决时（如 load-use 冒险），插入气泡

---

## 5. 控制冒险

### 5.1 产生原因

分支（branch）或跳转（jump）指令改变程序控制流。流水线在 ID 阶段才能判断分支是否跳转，但在 IF 阶段已经取入了下一条（可能错误的）指令。

### 5.2 示例

```
0x100: beq x1, x2, target    # 分支指令
0x104: add x3, x4, x5        # 下一条（可能被跳过）
0x108: sub x6, x7, x8
...
target: or x9, x10, x11      # 分支目标
```

在流水线中：

```
周期:   1      2      3      4      5
beq:   IF    ID    EX    MEM    WB
add:         IF    ID    ...  (flush: beq 跳转)
                ↑ beq 在 EX 阶段判断跳转
                  此时 add 已在 ID 阶段 → 需要冲刷
```

流水线在 EX 阶段才确定分支是否跳转，此时已取入了 1 条错误的指令（add）。如果分支判断更晚（如 MEM 阶段），错误指令会更多。

### 5.3 解决方法

- **静态分支预测**: predict-not-taken（继续执行下一条）、predict-taken（跳转到目标地址）
- **动态分支预测**: 使用分支历史表（Branch History Table）记录过去的分支行为
- **分支延迟槽**: RISC-V 等 RISC 架构中的分支指令后的指令槽
- **分支目标缓冲器**: 缓存分支目标地址

---

## 6. 冒险的代价

每次冒险都会引入**气泡**（bubble），即流水线中的空操作（NOP）：

```
正常流水线（无冒险）:
周期:  1    2    3    4    5    6
      I1   I2   I3   I4   I5   I6

一次停顿（插入一个气泡）:
周期:  1    2    3    4    5    6    7
      I1   I2   I3   ✗    I4   I5   I6
                    bubble
```

每个气泡意味着一个时钟周期内没有指令完成，CPI 因此增加。

---

## 7. 三类冒险对比总结

| 维度 | 结构冒险 | 数据冒险 | 控制冒险 |
|------|---------|---------|---------|
| 触发条件 | 资源争用 | 数据依赖 | 分支/跳转 |
| 典型频率 | 低（设计好可避免） | 高（程序普遍存在） | 中等（约 15-20% 指令是分支） |
| 解决成本 | 硬件增加（独立缓存） | 转发逻辑 + 一些停顿 | 预测器 + 冲刷逻辑 |
| CPI 影响 | 设计决定 | ~0.1-0.3 | ~0.2-0.5（无预测时） |
| 本模块重点 | 略 | ★★★ | ★★☆ |

---

## 总结

- **结构冒险**: 资源冲突 —— 增加资源解决
- **数据冒险**: 数据依赖 —— 转发 + 停顿解决
- **控制冒险**: 分支 —— 预测 + 延迟槽解决
- 所有冒险最终都可能导致**气泡**，降低 CPI
- 后续笔记分别深入数据冒险和控制冒险
