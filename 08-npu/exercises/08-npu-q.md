## 问题一：脉动阵列数据流追踪

### 问题描述

给定以下3×3权重矩阵 W 和 3×3 输入矩阵 X，请追踪通过一个 **3×3 的权重固定（Weight Stationary, WS）脉动阵列**的完整计算过程。输出矩阵 Y = W × X（矩阵乘法），即 Y[i][j] = Σ_k W[i][k] × X[k][j]。

```
        ┌          ┐         ┌          ┐
        │ 1  0  -1 │         │ 2  1  0  │
W =     │ 2  1   0 │    X =  │ 1  3  1  │
        │ 0 -1   1 │         │ 0  2  1  │
        └          ┘         └          ┘
```

**问题要求：**

1. 说明在WS数据流中，权重如何预加载到PE阵列中。
2. 绘制数据流图，标明每个时钟周期每个PE中的计算内容。
3. 展示输入矩阵 X 的元素如何流入阵列、部分和如何流动。
4. 给出完整的9个输出元素的值。
5. 对比如果使用输出固定（Output Stationary, OS）数据流，数据流动方式有何不同？

### 参考答案

#### (1) 权重预加载

在WS数据流中，每个PE预加载一个权重。具体加载方式为：
- PE(i, j) 预加载权重 W[i][j]。

因此，3×3 脉动阵列的权重加载为：
```
PE(0,0)=1  PE(0,1)=0  PE(0,2)=-1
PE(1,0)=2  PE(1,1)=1  PE(1,2)=0
PE(2,0)=0  PE(2,1)=-1 PE(2,2)=1
```

权重在计算过程中不移动，始终驻留在各自PE的本地寄存器中。

#### (2) 全局数据流模式

在WS脉动阵列中：
- **输入**：X 的列从左向右流动。第一个周期，X 的第一列进入第一列PE；第二个周期，第一列向右传递，同时第二列进入第一列PE。
- **部分和**：从上向下流动。每个PE累加本地的权重 × 进入的输入，然后将部分和传递给下方的PE。
- **输出**：从最下方一行的PE流出，得到输出矩阵 Y 的一列。

#### (3) 时钟周期追踪

为了便于追踪，将输入 X 的列向量记为：
```
X_col0 = [2, 1, 0]^T
X_col1 = [1, 3, 2]^T
X_col2 = [0, 1, 1]^T
```

**初始化**：所有PE的部分和寄存器 Psum = 0。

**时钟周期 1**：
- 输入 X_col0 进入阵列的第一列：x0 = 2（进入PE(0,0)）
- PE(0,0)：Psum += 1 × 2 = 2
- 其他PE无输入，保持空闲。

**时钟周期 2**：
- X_col0 从第一列向右移动到第二列：x0 = 2 进入PE(0,1)
- X_col1 进入第一列：x1 = 1 进入PE(0,0)
- PE(0,0)：Psum += 1 × 1 = 3
- PE(0,1)：Psum += 0 × 2 = 0
- PE(1,0)：PE(0,0)的部分和向下传递：Psum += 2 × 1 = 2（假设此时x1=1已到达PE(1,0)；实际时序需要根据具体设计中数据错开安排）

**注意**：在实际脉动阵列中，需要对输入数据进行"错开"（skewing）以保证正确的时序对齐。对于一个n×n的阵列，第j个输入列的到达需要相对于第j-1个输入列延迟一个周期进入第一行。因此，完整的数据流如下：

**重新以标准WS脉动阵列时序追踪（含skew）：**

定义：输入数据 X[k][j] 需要在正确的时钟进入第k行第0列的PE。

设时钟周期从 t=0 开始：

| 周期 | 进入PE(0,0) | PE(0,0)计算 | 进入PE(1,0) | PE(1,0)计算 |
|------|------------|-------------|------------|-------------|
| t=0  | X[0][0]=2  | 1×2=2      | -          | -           |
| t=1  | X[0][1]=1  | 3(累加)    | X[1][0]=1  | 2×1=2       |
| t=2  | X[0][2]=0  | 3(累加)    | X[1][1]=3  | 2+6=8       |
| t=3  | -          | -          | X[1][2]=1  | 8+2=10      |

对于第二行，输入总是比第一行晚1个周期到达（因为需要从上方向下流动）。

#### (4) 输出结果

直接计算验证：Y = W × X

```
Y[0][0] = 1×2 + 0×1 + (-1)×0 = 2
Y[0][1] = 1×1 + 0×3 + (-1)×2 = -1
Y[0][2] = 1×0 + 0×1 + (-1)×1 = -1
Y[1][0] = 2×2 + 1×1 + 0×0 = 5
Y[1][1] = 2×1 + 1×3 + 0×2 = 5
Y[1][2] = 2×0 + 1×1 + 0×1 = 1
Y[2][0] = 0×2 + (-1)×1 + 1×0 = -1
Y[2][1] = 0×1 + (-1)×3 + 1×2 = -1
Y[2][2] = 0×0 + (-1)×1 + 1×1 = 0
```

#### (5) OS与WS的对比

在输出固定（OS）数据流中：
- 输出部分和固定在PE中累积。
- 权重和输入都流动：权重沿一个方向流动，输入沿另一个方向流动。
- 每个PE对应一个输出元素，其内部的累积结果就是最终的Y[i][j]。
- OS的优势在于减少部分和的搬运，适合输出通道数大的场景。
- WS的优势在于权重固定，适合权重复用度高（如batch size大）的场景。

## 问题二：量化数学

### 问题描述

给定张量 T = [-1.5, 0.0, 3.2, 7.8, -0.5]（FP32），使用对称/非对称量化将其量化为INT8（有符号8位整数，范围[-128, 127]）。

**问题要求：**

1. 计算非对称量化（affine quantization）所需的 scale 和 zero_point。
2. 对每个元素进行量化（quantize）：T_q = round(T / scale) + zero_point。
3. 进行反量化（dequantize）：T' = (T_q - zero_point) × scale。
4. 计算量化-反量化过程的最大绝对误差（MAE）和均方误差（MSE）。
5. 如果使用对称量化（zero_point=0），scale 如何计算？非对称和对称量化的误差差异有多大？

### 参考答案

#### (1) 非对称量化的 scale 和 zero_point

INT8的范围：[q_min, q_max] = [-128, 127]。
张量的范围：[r_min, r_max] = [-1.5, 7.8]。

scale 的计算：
```
scale = (r_max - r_min) / (q_max - q_min) = (7.8 - (-1.5)) / (127 - (-128)) = 9.3 / 255 ≈ 0.03647
```

zero_point 的计算（取整）：
```
zero_point = round(q_min - r_min / scale) = round(-128 - (-1.5) / 0.03647)
= round(-128 - (-41.13)) = round(-86.87) = -87
```

另一种常用公式：zero_point = round(q_min - r_min / scale)，确保 r=0 精确映射。
zero_point = round(q_min - (-1.5) / 0.03647) = round(-128 + 41.13) = round(-86.87) = -87

验证：zero_point 应在 [q_min, q_max] 范围内，-87 ∈ [-128, 127]，符合要求。

#### (2) 量化过程

对每个元素进行量化：

```
T[0] = -1.5:  T_q[0] = round(-1.5 / 0.03647) + (-87) = round(-41.14) + (-87) = -41 - 87 = -128
T[1] = 0.0:   T_q[1] = round(0.0 / 0.03647) + (-87) = 0 - 87 = -87
T[2] = 3.2:   T_q[2] = round(3.2 / 0.03647) + (-87) = round(87.77) - 87 = 88 - 87 = 1
T[3] = 7.8:   T_q[3] = round(7.8 / 0.03647) + (-87) = round(213.94) - 87 = 214 - 87 = 127
T[4] = -0.5:  T_q[4] = round(-0.5 / 0.03647) + (-87) = round(-13.71) - 87 = -14 - 87 = -101
```

量化结果：T_q = [-128, -87, 1, 127, -101]

#### (3) 反量化过程

```
T'[0] = (-128 - (-87)) × 0.03647 = (-41) × 0.03647 = -1.495
T'[1] = (-87 - (-87)) × 0.03647 = 0 × 0.03647 = 0.0
T'[2] = (1 - (-87)) × 0.03647 = 88 × 0.03647 = 3.210
T'[3] = (127 - (-87)) × 0.03647 = 214 × 0.03647 = 7.805
T'[4] = (-101 - (-87)) × 0.03647 = (-14) × 0.03647 = -0.511
```

反量化结果：T' = [-1.495, 0.0, 3.210, 7.805, -0.511]

#### (4) 误差分析

| 原始值 | 反量化值 | 绝对误差 |
|--------|---------|---------|
| -1.5   | -1.495  | 0.005   |
| 0.0    | 0.0     | 0.000   |
| 3.2    | 3.210   | 0.010   |
| 7.8    | 7.805   | 0.005   |
| -0.5   | -0.511  | 0.011   |

最大绝对误差（MAE）= 0.011
均方误差（MSE）= (0.005² + 0² + 0.010² + 0.005² + 0.011²) / 5
= (0.000025 + 0 + 0.0001 + 0.000025 + 0.000121) / 5
= 0.000271 / 5 = 0.0000542

相对误差（以原始值的范围归一化）：0.011 / 9.3 ≈ 0.12%

#### (5) 对称量化对比

对称量化（zero_point = 0）：

scale_sym = max(|r_min|, |r_max|) / q_max
= max(1.5, 7.8) / 127 = 7.8 / 127 ≈ 0.06142

量化（对称）：
```
T_q[0] = round(-1.5 / 0.06142) = round(-24.42) = -24
T_q[1] = round(0.0 / 0.06142) = 0
T_q[2] = round(3.2 / 0.06142) = round(52.10) = 52
T_q[3] = round(7.8 / 0.06142) = round(127.00) = 127
T_q[4] = round(-0.5 / 0.06142) = round(-8.14) = -8
```

反量化（对称）：
```
T'[0] = -24 × 0.06142 = -1.474
T'[1] = 0
T'[2] = 52 × 0.06142 = 3.194
T'[3] = 127 × 0.06142 = 7.800
T'[4] = -8 × 0.06142 = -0.491
```

对称量化的最大绝对误差 = max(|-1.5+1.474|, |3.2-3.194|, |7.8-7.8|, |-0.5+0.491|) = max(0.026, 0.006, 0, 0.009) = 0.026

对称量化的误差（0.026）大于非对称量化（0.011），因为对称量化在值分布不对称时浪费了表示范围（负数侧的表示能力未充分利用）。对于高度不对称的数据（如ReLU后的全正激活值），非对称量化的优势更明显。

## 问题三：稀疏性格式对比

### 问题描述

在神经网络中，权重和激活值往往具有稀疏性（大量零值）。常见的稀疏性格式有**非结构化稀疏**（如90%稀疏度）和**结构化稀疏**（如NVIDIA的2:4结构化稀疏）。

**问题要求：**

1. 对于90%非结构化稀疏（即90%的权重为0），平均每4个元素中有几个非零元素？
2. 对于2:4结构化稀疏，每4个元素中强制恰好有2个非零元素。稀疏度是多少？
3. 解释为什么结构化稀疏在硬件实现上具有显著优势（对比非结构化稀疏）。
4. 如果使用CSR（Compressed Sparse Row）格式存储权重，估算90%非结构化稀疏和2:4结构化稀疏的存储压缩比。
5. 结合NVIDIA Ampere架构的Sparse Tensor Core，说明2:4结构化稀疏的实际加速原理。

### 参考答案

#### (1) 90%非结构化稀疏的非零元素数量

90%稀疏度意味着平均10%的元素非零。因此，每4个元素中，平均非零元素数量为：
4 × 10% = 0.4个

这意味着在90%稀疏度下，每4个元素中平均只有0.4个元素是有值的。非零元素的分布是完全随机的，不存在规律性。

#### (2) 2:4结构化稀疏的稀疏度

2:4结构化稀疏中，每4个元素有2个非零、2个为零。
稀疏度 = 零元素个数 / 总元素个数 = 2/4 = 50%。

2:4结构化稀疏的稀疏度只有50%，远低于非结构化稀疏的90%。然而，在硬件加速效果上，2:4结构化稀疏反而可能优于更高稀疏度的非结构化稀疏。

#### (3) 结构化稀疏的硬件优势

非结构化稀疏的硬件挑战：
- **不规则的内存访问**：非零元素的分布是随机的，加载权重时需要索引解码，导致不规则的内存访问模式，难以利用DRAM的burst传输特性。
- **负载不均**：不同处理单元处理的数据块稀疏度不同（有的块0个非零，有的块3个非零），导致PE利用率不均匀，部分PE闲置等待。
- **索引开销**：需要额外的元数据（如列索引、行指针）来标记非零元素的位置，占用了存储带宽和计算资源。
- **控制复杂**：稀疏矩阵乘法需要用非零元素索引动态调度计算，硬件控制逻辑复杂。

结构化稀疏的硬件优势：
- **规律的数据布局**：非零元素的位置是固定的（每4个中的前2个或后2个），硬件可以预知数据位置，不需要索引解码。
- **负载均衡**：每个块的非零元素数量固定（恰好2个），所有PE的负载完全相同。
- **高利用率**：2:4稀疏度的权重量减少一半，MAC阵列中对应的非零权重的计算可以成对合并，硬件利用率保持100%。
- **简化的控制逻辑**：不需要CSR/CSC索引解码，只需要一个简单的掩码（mask）指示哪些位置非零。

#### (4) 存储压缩比

**90%非结构化稀疏（使用CSR格式）：**

CSR格式需要存储：
1. 非零元素值（nnz个浮点数）
2. 列索引（nnz个整数，假设INT32）
3. 行指针（rows+1个整数，假设INT32）

假设权重矩阵为 N×N，总元素数 = N²，非零元素数 nnz = 0.1 × N²。

CSR存储量 = nnz × (4 + 4) + (N+1) × 4 = 0.1N² × 8 + 4N + 4

原始稠密存储量 = N² × 4

压缩比 ≈ (N² × 4) / (0.8N² + 4N) ≈ 当N足够大时：4/0.8 = 5× 压缩

**2:4结构化稀疏：**

只需要存储每4个元素中的2个非零元素，再加上一个2-bit的掩码（指示哪两个位置非零）。

存储量 = N² × 0.5 × 4 (半数值) + N² × 2/8 (掩码, bits转为bytes)
= 2N² + 0.25N² = 2.25N²

压缩比 = (4N²) / (2.25N²) ≈ 1.78×

2:4结构化的压缩比（1.78×）远低于90%非结构化的压缩比（约5×），但换来的是硬件效率和速度的大幅提升。

#### (5) NVIDIA Ampere架构的Sparse Tensor Core原理

NVIDIA Ampere架构（A100）引入了2:4结构化稀疏的支持：

1. **训练阶段**：先正常训练稠密模型 → 使用渐进式剪枝（gradual pruning）微调，使权重自然收敛到2:4模式 → 利用重训练恢复精度损失。

2. **推理阶段**：权重以2:4模式存储（每4个存2个，加2-bit掩码）。加载时，非零权重配对送入Sparse Tensor Core。Sparse Tensor Core将两个非零权重分别与对应的输入相乘并累加，在一个周期内完成原本需要两个周期的计算。

3. **加速效果**：理论上，权重减半使MAC操作减半，计算速度翻倍。在实际测试中，A100的稀疏矩阵乘法达到稠密矩阵乘法的2×吞吐量。

4. **精度影响**：在CV和NLP任务上，2:4结构化稀疏通常能将精度损失控制在0.1%-0.5%以内，远优于同等稀疏度的非结构化剪枝导致的精度下降。

## 问题四：INT8与FP32的能量效率对比

### 问题描述

估计对一个1024×1024的矩阵乘法（C = A × B，全部从DRAM读取）使用INT8相比FP32可以节省多少能量。一次矩阵乘法需要执行1024³ = 约1.07×10⁹次MAC运算（严格来说是N³次乘加，但每行每列各计算一次点积，也可看作N³次乘+ N³次加，或约2N³次总操作）。

**能量假设**（基于Horowitz ISSCC'14的45nm CMOS数据，简化为比例模型）：
- 一次MAC运算能量（FP32）：1 unit（基准）
- 一次MAC运算能量（INT8）：0.2 unit（INT8乘法器面积小，能耗约为FP32的20%）
- 一次SRAM访问（32-bit）：10 units
- 一次DRAM访问（32-bit）：200 units
- 一次SRAM访问（8-bit）：6 units（估算）
- 一次DRAM访问（8-bit）：150 units（估算）

**注意**：INT8的数据搬运带宽需求是FP32的1/4。

**问题要求：**

1. 分别计算FP32和INT8执行1024×1024矩阵乘法的总能量（包括计算能量和数据搬运能量）。
2. 假设数据重用策略：权重和输入都只从DRAM读取一次，结果写回DRAM一次（即不考虑tiling优化）。
3. 计算INT8相比FP32的能量节省比例。
4. 讨论如果加入tiling（将矩阵分块到片上SRAM中），能量节省比例会如何变化。
5. 思考：为什么在实际NPU中，INT8的能效提升往往达不到理论最优值？

### 参考答案

#### (1) 总能量计算（无tiling优化）

**FP32场景：**

MAC操作总数：1024³ = 1,073,741,824 ≈ 1.07×10⁹ 次MAC

计算能量：1.07×10⁹ × 1 = 1.07×10⁹ units

数据搬运：
- 权重矩阵A：1024×1024 = 1,048,576 个FP32值 → 从DRAM读取一次
- 输入矩阵B：1024×1024 = 1,048,576 个FP32值 → 从DRAM读取一次
- 结果矩阵C：1024×1024 = 1,048,576 个FP32值 → 写回DRAM一次（读/写能耗近似，均计为DRAM访问）

DRAM总访问次数（以32-bit word计）：3 × 1,048,576 = 3,145,728

DRAM总能耗：3,145,728 × 200 = 629,145,600 units

FP32总能量 = 计算能量 + 数据搬运能量 = 1.07×10⁹ + 6.29×10⁸ = 1.70×10⁹ units

其中数据搬运占比：6.29×10⁸ / 1.70×10⁹ = 37%

**INT8场景：**

MAC操作总数：1.07×10⁹ 次（操作数相同，但每次运算为INT8）

计算能量：1.07×10⁹ × 0.2 = 2.15×10⁸ units

数据搬运（注意INT8每元素1字节，总数据量为FP32的1/4）：
- 权重矩阵A：1024×1024 = 1,048,576 个INT8值 → 1,048,576字节
- 输入矩阵B：1024×1024 = 1,048,576 个INT8值 → 1,048,576字节
- 结果矩阵C：1024×1024 = 1,048,576 个INT8值 → 写回DRAM

DRAM总访问次数（以byte计）：3 × 1,048,576 = 3,145,728 字节

DRAM总能耗：3,145,728 × 150/4 = 3,145,728 × 37.5 = 117,964,800 units

**注意**：8-bit DRAM访问能量为150 units/32-bit word，相当于37.5 units/8-bit。

INT8总能量 = 2.15×10⁸ + 1.18×10⁸ = 3.33×10⁸ units

#### (2) INT8 vs FP32能量对比

| 项目 | FP32 | INT8 | 节省比例 |
|------|------|------|---------|
| 计算能量 | 1.07×10⁹ | 2.15×10⁸ | 79.9% |
| 数据搬运能量 | 6.29×10⁸ | 1.18×10⁸ | 81.2% |
| 总能量 | 1.70×10⁹ | 3.33×10⁸ | 80.4% |

INT8总能量为FP32的 3.33×10⁸ / 1.70×10⁹ ≈ 19.6%，即**节省约80.4%的能耗**。

#### (3) 加入tiling后的分析

如果使用tiling，将矩阵分块到片上SRAM中：

假设SRAM大小为256KB，对于FP32（4字节/元素），可缓存约256K/4 = 64K元素，即256×256的tile。

对于INT8（1字节/元素），同样大小的SRAM可缓存256K个元素，即512×512的tile。

FP32场景（带tiling）：
- DRAM访问：每个tile加载A和B的子块，SRAM中完成多次MAC后写回C的子块。
- 假设tile大小256，需要 (1024/256)² = 16 个tiles。
- 每个tile加载A的256×1024行切片、B的1024×256列切片，写回C的256×256子块。
- DRAM总访问：2 × 1024³/256 + 1024² = 2 × 4,194,304 + 1,048,576 = 9,437,184 words
- DRAM能耗：9,437,184 × 200 = 1.89×10⁹ → 反而比无tiling高？不对，因为tiling减少了逐点DRAM访问，但这里的分块策略不对。

实际上，正确的tiling分析是：
- 对于MAC操作，每个DRAM加载的A和B元素可以被多次使用。
- 最优tiling：DRAM访问次数可以减少到约 2 × N³/B + N²/B（其中B为tile能容纳的元素数）。
- 对于FP32（tile=256），DRAM加载约 2 × 1024³/256 = 8,388,608 words
- 对于INT8（tile=512），DRAM加载约 2 × 1024³/512 = 4,194,304 words

使用tiling后，INT8的计算和数据搬运总能量仍然显著低于FP32，且INT8由于数据密度高，在同等SRAM容量下可以容纳更大的tile，进一步减少DRAM访问次数。

#### (4) 实际INT8能效不如理论值的原因

1. **量化精度损失需要补偿**：INT8量化可能导致模型精度下降，往往需要量化感知训练（QAT）或混合精度来恢复精度，这增加了开发成本。
2. **激活值量化复杂性**：激活值的范围逐层变化，需要动态量化（dynamic quantization）或校准数据集来统计范围，增加了硬件复杂度和预处理开销。
3. **INT8计算单元的利用率问题**：当网络层的小尺寸操作（如小矩阵）无法充分填满SIMD单元时，INT8的吞吐量优势打折扣。
4. **带宽瓶颈转移**：虽然数据搬运量减为1/4，但计算速度更快后，I/O带宽可能成为新的瓶颈。
5. **累加器精度要求**：INT8 MAC累加时需要更高的精度（通常INT32累加器），这部分存储和能量开销在理论模型中被低估了。

## 问题五：AI编译器优化——算子融合

### 问题描述

给定以下神经网络层的计算序列：

```
输入特征图 → Conv (3×3卷积) → ReLU → Add (逐元素加法，加上残差连接) → Pool (2×2最大池化) → 输出特征图
```

**问题要求：**

1. 解释算子融合（operator fusion / kernel fusion）的基本原理和目标。
2. 分析上述序列中哪些算子可以融合，哪些不能，并说明原因。
3. 在不融合的情况下，描述内存访问流程（包括中间结果存取）。
4. 在融合后，描述融合后的计算流程，并计算内存节省量（假设输入特征图为C×H×W，C=64, H=W=56，所有数据为FP32）。
5. 以TVM或XLA为背景，说明算子融合在AI编译器中的实现机制（如TVM的融合规则）。

### 参考答案

#### (1) 算子融合的基本原理

算子融合（Operator Fusion）是AI编译器中的一项关键优化技术，其核心思想是：**将多个连续的计算算子合并为一个复合算子（kernel），在单个硬件内核中一次性执行完毕，避免中间结果在全局内存（DRAM）和计算单元之间反复搬运。**

融合的基本原理：
- **减少内存带宽压力**：如果每个算子独立执行，产生的结果必须写回DRAM，下一个算子再读回。融合后，中间结果只存在于片上的寄存器或SRAM缓存中，不需要DRAM访问。
- **改善数据局部性**：融合后的内核在时间上更紧凑，数据的生产和消费之间的时间间隔缩短，缓存命中率提高。
- **减少内核启动开销**：GPU等硬件上启动一个kernel有固定开销（~微秒级），融合减少了总kernel数量，节省了这部分开销。

融合的目标是找到算子之间的**生产者-消费者关系**，将消费者算子的计算"内联"到生产者算子中。

#### (2) 算子融合分析

给定序列：Conv → ReLU → Add → Pool

**可以融合的部分：**

- **Conv + ReLU 可以融合**：每个卷积输出像素计算完成后，立即施加ReLU（f(x) = max(0, x)）。ReLU是元素级操作（element-wise），不改变数据维度，且计算模式极为简单（一次比较），融合后几乎没有额外开销。融合规则：**Element-wise算子可以与前一个算子融合**。

- **Conv+ReLU+Add 也可以融合**：如果残差连接（Add的另一个输入）在融合前已准备好，那么可以在卷积+ReLU的计算循环中，将ReLU输出与残差输入直接相加。Add也是逐元素操作，融合成本很低。

- **Conv+ReLU+Add+Pool 有限制的融合**：池化操作是一个**降采样（downsampling）**操作（2×2最大池化将HW各减半）。它可以在Add之后立即执行，因为池化的计算不依赖于输出空间中相邻块之外的数据（pooling window是2×2的局部区域）。因此，Conv+ReLU+Add+Pool可以融合为一个kernel。

**不能融合的情况（一般规则）：**

- **Reshape/Transpose类操作**：改变数据布局的算子不能轻易融合，因为它们破坏了数据的连续性。
- **Broadcast操作**：当张量形状不匹配时，广播规则增加了融合的复杂性。
- **输出形状不同的算子**：如果两个算子的输出张量形状差异过大，融合后的线程映射（thread mapping）变得困难。

在本例中，所有算子都可以融合为单个kernel。

#### (3) 不融合时的内存访问流程

不融合时，每个算子独立执行：

```
1. Conv kernel:
   - 读入：输入特征图 (64×56×56) + 卷积权重 (3×3×64×64)
   - 写入：卷积输出 (64×56×56)
   → DRAM写：64×56×56×4 = 802,816 字节 ≈ 0.77 MB

2. ReLU kernel:
   - 读入：卷积输出 (64×56×56)
   - 写入：ReLU输出 (64×56×56)
   → DRAM写：0.77 MB

3. Add kernel:
   - 读入：ReLU输出 (64×56×56) + 残差输入 (64×56×56)
   - 写入：Add输出 (64×56×56)
   → DRAM写：0.77 MB

4. Pool kernel:
   - 读入：Add输出 (64×56×56)
   - 写入：池化输出 (64×28×28)
   → DRAM写：64×28×28×4 = 200,704 字节 ≈ 0.19 MB
```

**总DRAM写入量**：0.77 + 0.77 + 0.77 + 0.19 = 2.5 MB （不含输入读取）

**总DRAM读取量**：
- Conv输入：0.77 MB + 权重：3×3×64×64×4 = 147,456字节 ≈ 0.14 MB
- ReLU输入：0.77 MB
- Add输入：0.77 MB（ReLU输出）+ 0.77 MB（残差）
- Pool输入：0.77 MB

合计 ≈ 0.77 + 0.14 + 0.77 + 0.77 + 0.77 + 0.77 = 3.99 MB（读取）

**总DRAM访问量（读+写）**：3.99 + 2.50 ≈ 6.49 MB

#### (4) 融合后的计算流程与内存节省

**融合后的计算流程：**

```
Fused kernel (Conv + ReLU + Add + MaxPool):
  for (oh in 0..H_out)      // 输出高度（池化后28）
    for (ow in 0..W_out)    // 输出宽度（池化后28）
      for (oc in 0..C_out)  // 输出通道64
        sum = 0
        for (kh in 0..2)
          for (kw in 0..2)
            for (ic in 0..C_in)
              sum += input[oh*2+kh][ow*2+kw][ic] × weight[oc][kh][kw][ic]
        // ReLU融合
        sum = max(0, sum)
        // Add融合
        sum = sum + residual[oh][ow][oc]  // 注意池化前残差对齐
        // MaxPool融合：取2×2窗口的最大值
        pooled = max(pooled, sum)  // 在内部循环累积
        将pooled写入输出特征图
```

融合后，**中间结果（Conv输出、ReLU输出、Add输出）不再写回DRAM**。它们在PE的本地寄存器或片上SRAM中直接传递。

**内存节省计算：**

融合后，只需要：
- 从DRAM读入：输入特征图、权重、残差输入
- 写入DRAM：最终池化输出

**融合后DRAM写入量**：池化输出 × 4 = 64 × 28 × 28 × 4 = 200,704 字节 ≈ 0.19 MB

**融合后DRAM读取量**：
- 输入特征图：0.77 MB
- 权重：0.14 MB
- 残差输入：0.77 MB

合计 ≈ 1.68 MB

**总DRAM访问量（读+写）**：1.68 + 0.19 = 1.87 MB

**内存节省**：
- 相比不融合的6.49 MB，融合后为1.87 MB，减少了约 **71%** 的DRAM访问量。
- 更关键的是，**消除了3个中间张量的DRAM读写**（Conv输出、ReLU输出、Add输出），每个0.77 MB。

#### (5) TVM/XLA中的算子融合实现

**TVM的实现：**

TVM的算子融合通过**计算图优化**和**融合规则引擎**实现，核心步骤：

1. **构建计算图**：将网络解析为计算图（DAG），节点为算子，边为张量依赖。

2. **分组（Grouping）**：从输出节点开始反向遍历，应用融合规则判断是否可以合并。TVM定义了几种融合模式：
   - **Element-wise融合**：将连续的逐元素算子（如ReLU、Add、Sigmoid）融合为一个kernel。
   - **广播融合**：将带有广播的逐元素操作与上游算子融合。
   - **Injective融合**：将单射（injective，即一个输出对应唯一一个输入）的算子融合。
   - **降采样融合**：如Conv2D + Pooling的融合。

3. **融合规则**：TVM使用基于支配树（dominator tree）的分析来确定融合是否合法。核心规则是：
   - 下游算子必须是**单射**（element-wise）或**降采样**（reduce），才能与上游融合。
   - 如果下游算子有多个输入（如Add），其中一个输入必须来源于上游融合组，另一个可以是外部输入。

4. **代码生成**：融合后的组传递给tensor expression（TE）编译器，生成统一的CUDA/OpenCL代码。TVM的auto-tuning框架（AutoTVM/AutoScheduler）可以对融合kernel的循环分块、展开因子等进行自动调优。

**XLA（Accelerated Linear Algebra）的实现：**

XLA通过HLO（High-Level Optimizer）图优化实现融合：
1. **HLO图的构建与遍历**：寻找符合融合模式的子图。
2. **融合指令生成**：生成名为`kFusion`的特殊指令，包含子图中的所有操作。
3. **Layout优化**：在融合过程中考虑数据布局的优化，避免不必要的转置。
4. **Buffer分配**：融合kernel的输入和输出使用别名分析（alias analysis），最小化中间buffer的分配。

TVM和XLA的一个重要差异是：TVM使用基于规则的显式融合策略，开发者可以编写自定义融合规则；XLA使用基于编译器的自动融合，对用户透明但规则固定。
