# 02 — 乱序执行（Out-of-Order Execution）

## 概述

### 为什么需要乱序执行？

在有序（in-order）超标量处理器中，如果指令 A 因为等待数据而停顿，它后面所有指令都会被阻塞——即使这些指令完全独立于 A。

```
ADD X1, X2, X3     ; 需要等待 X2（假设 X2 来自缓存缺失）
SUB X4, X5, X6     ; 完全独立于 X1，却必须等待
MUL X7, X8, X9     ; 完全独立
```

浪费大量发射槽。

**乱序执行的核心思想**：指令准备好操作数就执行，提交（commit）时再按程序顺序提交。这打破了指令的执行顺序，但保持了程序的提交顺序，从而保证了程序的正确性。

## 乱序执行的基本流程

现代乱序执行处理器的四条基本原则（有时分为四步）：

1. **取指（Fetch）**：从指令缓存中取出指令，按程序顺序
2. **发射（Issue/Dispatch）**：将指令发送到功能单元等待队列，当操作数就绪时执行
3. **执行（Execute）**：功能单元执行指令，结果可以转发给其他指令
4. **提交（Commit/Retire）**：按程序顺序将结果写回寄存器或内存。乱序执行的结果在提交前是"推测性的"

> 核心要点：取指和提交按顺序，执行可以乱序。

## 三大关键结构

### 1. 重排序缓冲（Reorder Buffer, ROB）

**功能**：ROB 是一个循环缓冲（circular buffer），按程序顺序记录所有在途（in-flight）指令的信息。当指令执行完成时，结果暂时保存在 ROB 中；当指令到达 ROB 的头部时，才正式提交。

**ROB 的作用**：
- **按序提交**：保证程序正确性（精确异常、按序写回）
- **保存推测状态**：分支误预测后，可以快速清空 ROB 中误路径上的指令
- **寄存器的临时快照**：提供寄存器重命名的存储空间

**ROB 条目典型格式**：

| 字段 | 含义 |
|------|------|
| 指令类型 | 算数/加载/存储/分支 |
| 目标寄存器 | 写入的寄存器号 |
| 结果值 | 执行完成后的值 |
| 就绪标志 | 结果是否已计算完成 |
| 异常标志 | 执行过程中是否发生异常 |
| 分支预测标志 | 是否位于推测路径上 |

**ROB 的容量**：44 ~ 512+ 条指令不等。

| CPU | ROB 条目数 |
|-----|-----------|
| Intel Core (1st gen) | 128 |
| Intel Golden Cove | 512 |
| AMD Zen 4 | 320 |
| Apple M1 Firestorm | 630+ |

### 2. 保留站（Reservation Station, RS）

**功能**：保留站是功能单元前的等待队列。每条指令放在保留站中，等待其所有操作数就绪。当操作数就绪且功能单元有空时，指令发射执行。

**保留站的工作方式**（以 Tomasulo 算法为基础）：

1. 译码后的指令进入保留站
2. 保留站中维护操作数就绪标志
3. 操作数可以是已就绪的值，也可以是 **指向 ROB 结果的标签（Tag）**
4. 当功能单元输出结果时，通过公共数据总线（CDB, Common Data Bus）广播给所有保留站
5. 保留站监听 CDB，如果某条指令等待的标签已被广播，则获取结果并标记为就绪
6. 当所有操作数就绪且功能单元空闲时，指令发射执行

**集中式 vs 分布式保留站**：

- 集中式：所有功能单元共享一个保留站（调度灵活性高，但硬件复杂）
- 分布式：每个功能单元有自己的保留站（设计简单，但灵活性低）

### 3. 寄存器重命名（Register Renaming）

**目的**：消除 WAR（Write After Read）和 WAW（Write After Write）冲突。

```
// 代码：有 WAR 冲突
MUL R1, R2, R3   // 读 R3
ADD R3, R4, R5   // 写 R3 (WAR: 实际 ADD 可以在 MUL 读取 R3 后立即写)
```

这两种冲突是"名称相关"（name dependence），而不是真正的数据相关。通过重命名，可以消除这些虚假依赖。

**如何工作**：
- 处理器维护一个寄存器映射表（Register Alias Table, RAT），将架构寄存器（如 R1, R2）映射到物理寄存器（如 P32, P45）
- 每条写入架构寄存器的指令会分配一个新的物理寄存器
- 后续读取该架构寄存器的指令会自动读取到最新的物理寄存器
- 旧的物理寄存器在不再被需要时释放

**重命名前后的数据流**：

```
// 原始代码
MUL R1, R2, R3
ADD R3, R4, R5   // WAR on R3

// 重命名后（物理寄存器 P0, P1, P2, P3...）
MUL P32, P30, P31  // R1 → P32, R2 → P30, R3 → P31 (当前值)
ADD P33, P34, P35  // R3 → P33 (新分配), R4 → P34, R5 → P35
// 两条指令完全独立，可以乱序执行
```

## Tomasulo 算法（概念性理解）

Robert Tomasulo 在 1967 年（IBM System/360 Model 91）提出了一种硬件算法，允许指令乱序执行。这是现代所有乱序处理器的理论基础。

### 算法的三个核心机制

1. **保留站**：指令等待操作数就绪，无需中央调度器
2. **寄存器重命名**：通过保留站中的标签机制消除名称相关
3. **公共数据总线（CDB）**：计算结果广播给所有等待的指令

### 简单示例

```
L.D F0, 0(R1)      // 加载 F0
ADD.D F4, F0, F2   // 等待 F0 → 相关
MUL.D F6, F8, F10  // 独立 → 可以先执行
```

执行顺序（理想情况）：
1. `L.D` → 加载（缓存命中，2 周期）
2. `MUL.D` → 操作数就绪，即使排在后面也可以先执行（4 周期）
3. `ADD.D` → 需等待 F0 就绪
4. 提交：按 `L.D → ADD.D → MUL.D` 顺序提交

## 乱序执行的整体数据流

```
┌──────────┐    ┌───────────┐    ┌────────────────────┐
│ 取指（按序）│ → │ 译码+重命名 │ → │ 分派到保留站 / ROB │
│          │    │ (合并步)   │    │                    │
└──────────┘    └───────────┘    └────────┬───────────┘
                                         │
                                         ▼
                             ┌───────────────────┐
                             │ 保留站（等待操作数） │
                             │ 功能单元（乱序执行） │
                             └────────┬──────────┘
                                      │ (CDB 广播)
                                      ▼
                             ┌───────────────────┐
                             │    ROB（按序提交）   │
                             │ → 寄存器写回        │
                             │ → 内存写回          │
                             └───────────────────┘
```

## 精确异常（Precise Exceptions）

乱序执行的一个关键挑战：如果一条指令触发异常，此时它后面的指令可能已经执行完毕。如何保证异常时的状态一致？

- 异常指令之前的所有指令必须已提交（结果已写入寄存器/内存）
- 异常指令之后的所有指令必须"看起来"从未执行过（flush ROB 中该条目之后的所有指令）
- ROB 保证了精确异常的实现

## 乱序执行的关键设计考量

1. **ROB 大小**：更大的 ROB 能容纳更多的在途指令，有利于找到并行性，但增加了延迟和功耗
2. **保留站数量**：影响调度灵活性
3. **物理寄存器的数量**：影响重命名的深度
4. **CDB 广播带宽**：影响结果分发速率
5. **提交宽度（Commit Width）**：每周期最多提交的指令数
