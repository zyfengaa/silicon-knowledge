# 01 — CPU vs GPU 设计哲学

> CPU 为延迟优化，GPU 为吞吐优化。理解两者在设计目标上的根本差异，是理解 GPU 架构的起点。

---

## 1. 设计目标差异

### CPU：延迟优化（Latency-Optimized）

CPU 的设计目标是**让单条指令尽可能快地执行完毕**。为此，CPU 投入了大量晶体管和功耗用于：

- **超标量乱序执行（Out-of-Order Execution）**：通过重排序缓冲区（ROB）和保留站（Reservation Stations）来发掘指令级并行（ILP），尽可能在每个周期填满执行单元。
- **分支预测（Branch Prediction）**：现代 CPU 的预测准确率超过 95%，大幅减少分支误预测带来的流水线冲刷代价。
- **大容量缓存**：L1/L2/L3 三级缓存结构，L3 可达数十 MB，以降低访存延迟。
- **深度流水线**：x86 CPU 通常有 14-20 级流水线，以提升时钟频率。

这些机制的共同代价是**高复杂度、高功耗、大芯片面积**，但换来了出色的**单线程性能**。

| CPU 特性 | 目的 | 代价 |
|---------|------|------|
| 乱序执行 | 利用 ILP 隐藏延迟 | 功耗 +30%，面积 +20% |
| 分支预测 | 减少分支惩罚 | 预测器占用 ~10% 核心面积 |
| 大缓存 | 降低平均访存延迟 | L3 缓存占用 ~50% 芯片面积 |
| 深度流水线 | 提高时钟频率 | 分支误预测惩罚更大 |

### GPU：吞吐优化（Throughput-Optimized）

GPU 的设计目标是**让单位时间内执行尽可能多的线程**。GPU 不追求单条指令的快速完成，而是：

- **大量简单核心**：每个 CUDA Core 只执行简单的 FP32/INT32 运算，没有复杂的乱序逻辑。
- **线程级并行（TLP）来隐藏延迟**：当某个 warp 因访存而等待时，调度器立即切换到另一个就绪的 warp，计算单元始终保持忙碌。
- **极小的缓存**：GPU 缓存主要用于暂存数据和减少显存带宽压力，而非降低单次访存延迟。
- **宽 SIMT 执行模型**：32 个线程组成一个 warp，共享一个取指/译码单元，最大化控制逻辑的利用率。

| GPU 特性 | 目的 | 优势 |
|---------|------|------|
| 数千个并发线程 | 用 TLP 隐藏延迟 | 无需复杂的乱序逻辑 |
| 大量 FP32 ALU | 提高浮点吞吐 | ~80 TFLOPS（H100） |
| 小缓存/高带宽 | 匹配并行访存模式 | HBM 达 3.35 TB/s |
| 简单有序核心 | 降低面积和功耗 | 更多核心/芯片面积 |

## 2. 延迟隐藏（Latency Hiding）

### 核心思想

延迟隐藏是 GPU 最关键的设计概念之一。当某个 warp 发出访存请求时，结果需要数百个周期才能返回。与其让执行单元空闲等待，GPU 利用**零开销的线程切换**（zero-overhead context switch）切换到另一个就绪的 warp。

```
时间轴 (周期):
          
Warp 0: |计算|计算|    等待内存     |计算|计算|
Warp 1: |   |计算|计算|    等待内存     |计算|
Warp 2: |   |   |计算|计算|计算|    等待内存  |
Warp 3: |   |   |   |计算|计算|计算|          |

调度器: |←─── 4 个 warp 交替执行，计算单元始终忙碌 ──→|
```

每个 warp 都有自己独立的寄存器文件上下文。切换 warp 只需要改变 warp ID 和对应的寄存器指针，不需要像 CPU 那样保存/恢复堆栈，因此切换开销**几乎为零**。

### 需要多少线程才能隐藏延迟？

隐藏延迟所需的线程数与存储器延迟、线程指令吞吐量有关：

```
所需活动 warp 数 ≈ 存储器延迟（周期）÷ 线程间指令发射间隔
```

例如 DRAM 延迟 ~400 周期，SM 每周期可发射指令，则至少需要 400 个 warp（~12800 线程）才能完全隐藏延迟。这正是 GPU 设计为支持数万个并发线程的原因。

### CPU 与 GPU 的隐藏延迟策略对比

| 策略 | CPU | GPU |
|------|-----|-----|
| 隐藏什么延迟 | 缓存未命中（~10-100 周期） | 显存访问（~400-800 周期） |
| 如何隐藏 | 乱序执行 + 推测执行 | 大量线程交替执行 |
| 需要的并行度 | 指令级并行（ILP） | 线程级并行（TLP） |
| 切换开销 | 数百周期（上下文切换） | 零（硬件多线程） |

## 3. 峰值算力对比

### 理论峰值 FLOPS

现代 CPU 与 GPU 在浮点算力上的差距高达 **40 倍以上**：

| 处理器 | FP32 TFLOPS | FP64 TFLOPS | TDP | 工艺 |
|--------|-------------|-------------|-----|------|
| Intel Xeon Platinum 8490H (Sapphire Rapids) | ~2.0 | ~1.0 | 350W | Intel 7 |
| AMD EPYC 9654 (Genoa) | ~3.0 | ~1.5 | 360W | TSMC 5nm |
| Apple M2 Ultra | ~3.4 | ~0.3 | 140W | TSMC 5nm |
| **NVIDIA H100 SXM** | **~67** | **~34** | **700W** | **TSMC 4N** |
| **NVIDIA RTX 4090** | **~82** | **~1.3** | **450W** | **TSMC 4N** |
| **AMD MI300X** | **~131** | **~65** | **750W** | **TSMC 5nm/6nm** |

**关键观察**：

- GPU 在 FP32 上比 CPU 有 **20-40 倍**的算力优势
- GPU 的 FP64 算力因架构而异（H100 保留 1:2 的 FP64:FP32 比例，RTX 4090 仅 ~1:64）
- GPU 的能效比（TFLOPS/Watt）同样远高于 CPU

### 为什么会有这么大的差距？

根本原因在于**晶体管用途的分配**：

```
CPU 芯片面积分配（大致）：
┌─────────────────────────────────────────────┐
│   L3 缓存            │  控制逻辑            │
│   (~50%)             │  (OOO + 分支预测)    │
│                      │   (~25%)             │
├──────────────────────┴──────────────────────┤
│   计算单元 (ALU/FPU)  │  内存控制器 + I/O   │
│   (~15%)             │   (~10%)             │
└─────────────────────────────────────────────┘

GPU 芯片面积分配（大致）：
┌─────────────────────────────────────────────┐
│               计算单元 (CUDA Core / ALU)     │
│               (~70%)                        │
├─────────────────────────────────────────────┤
│   共享内存 + L1  │   控制逻辑 (简单)        │
│   (~15%)         │   (~5%)                  │
├─────────────────────────────────────────────┤
│   内存控制器 + I/O   │  寄存器文件           │
│   (~5%)             │   (~5%)               │
└─────────────────────────────────────────────┘
```

CPU 花费 ~75% 的面积在缓存和控制逻辑上，而 GPU 将 ~70% 的面积用于计算单元。同样的芯片面积和功耗预算下，GPU 自然拥有更多的计算核心。

## 4. 适用范围

### CPU 的强项

- 分支密集的代码（编译器、操作系统、数据库）
- 延迟敏感的应用（交互式系统、实时控制）
- 串行逻辑多、并行度低的计算
- 单线程性能至关重要的场景

### GPU 的强项

- 数据并行计算（矩阵乘法、卷积、向量运算）
- 计算密集、分支简单的任务（深度学习训练/推理）
- 大规模并行处理（图像渲染、物理模拟）
- 吞吐优先的工作负载（批处理推理、科学计算）

### 异构计算

现代高性能系统通常采用 CPU + GPU 异构架构，让 CPU 处理控制密集的串行部分，GPU 处理计算密集的并行部分。

```
典型异构执行流程：
1. CPU 端初始化数据、分配内存
2. CPU → GPU 传输数据（通过 PCIe 或 NVLink）
3. GPU 执行大规模并行 Kernel
4. GPU → CPU 传回结果
5. CPU 汇总结果并进行后续处理
```

这种模式同时利用了 CPU 的低延迟和 GPU 的高吞吐，是高性能计算和深度学习系统的基础架构模式。
